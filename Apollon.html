Apollon

Apollon est une application web de bases de connaissances fiscales, RH et retraites destinée aux agents de la DGFiP. Son développement s'est fait avec le CMS Drupal. Ma participation à ce projet a été de gérer des tickets d'anomalie via le système Jira, et le dépôt de code se faisait via GitLab. Le développement, suivant la méthode agile, impliquait une réunion quotidienne d'une quinzaine de minutes, à laquelle je participais, organisée afin de présenter les avancées et les poursuites de chacun.

Présentation de Jira/Confluence

Jira est l'outil de gestion de tickets sous la forme d'un kanban utilisé à la DGFiP, en parallèle de Confluence qui permet de fournir de la documentation sur les projets en cours.

Chaque ticket était traité selon un workflow bien définie :

  • Nouveau : Une fois qu'un ticket est créé, il est évalué par le développeur en charge du ticket afin de reproduire l'anomalie et de cerner d'où vient le problème. Il passe donc en "Évaluation".
  • Evaluation : Une fois évalué, le problème commence à être traité par la personne en charge. Il passe donc à "En cours".
  • En cours : Si le développeur ne trouve pas de solution, il repasse le ticket à évaluer afin de mieux cerner le problème. Dans le cas contraire, il passe le ticket à tester en "Recette".
  • Recette : Une fois en recette, l'anomalie est de nouveau testée afin de voir si elle a été résolue ou non. Si l'anomalie est toujours présente, la recette est dite KO et le ticket repasse à "Évaluation". Mais si l'anomalie a bien été résolue, la recette est OK et le ticket passe en "Résolu".
  • Résolu : Une fois résolu, le ticket est fermé et passe à "Fermé".
  • Fermé : Si l'anomalie réapparaît, le ticket peut être réouvert et repassé à "Nouveau".
  • Bloqué : À chaque étape, le ticket peut être passé à bloqué si l'anomalie ne peut être résolue de suite. Il faudra par la suite réouvrir le ticket pour le faire repasser à "Nouveau".
  • Abandonné : À toutes les étapes, le ticket peut être abandonné s'il n'est plus jugé pertinent ou résolu par une autre anomalie corrigée.
GitLab

Le code du projet était hébergé sur le système de gestion de versions GitLab propre à la DGFiP. Lorsqu'un code devait être modifié, le développeur créait une branche locale basée sur celle de développement avec comme nom le numéro du ticket, effectuait ses modifications, puis réalisait un commit explicite suivi d'un push et d'une demande de fusion (merge request). Ensuite, selon les disponibilités, un autre développeur effectuait une revue de code et validait ou non la demande de fusion.
